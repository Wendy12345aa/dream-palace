# Chapter 5 — UI / UX Bible

> *The best interface is one that disappears into the world.*

---

# Purpose

The Character Bible defines who the companions are.

The UI / UX Bible defines how players interact with them.

Dream Palace should never feel like a dashboard.

Players should feel that they are speaking with trusted advisors inside a living kingdom.

Every interface should strengthen immersion rather than interrupt it.

---

# UI Philosophy

Dream Palace does not separate gameplay from interface.

The interface is part of the world.

Every major interaction should feel like a conversation instead of opening another menu.

The player should always feel present inside the kingdom.

The goal is not to make beautiful panels.

The goal is to make the player forget they are using panels at all.

---

# Core Design Principles

## Character-Owned Interface

Every important system belongs to someone.

| Function | Companion | Interface Form |
|----------|-----------|----------------|
| Treasury | MP | Ledger, budget report, resource allocation |
| Technology | Kel | Blueprints, mechanisms, prototypes |
| Quality | Tab | Documents, reviews, approvals, audit notes |
| Investigation | CX | Maps, reports, hidden evidence, intelligence board |
| Culture | Mika | Sketches, murals, city identity, visual plans |
| Events | Vincey | Announcements, festivals, ceremonies, public mood |
| Planning | Shian | Strategy table, city models, routes, long-term plans |

Players should remember people before icons.

---

## Context Before Menu

Players should discover information naturally.

Example:

Morning Court

↓

MP reports a food shortage

↓

Player asks MP for details

↓

Treasury interface appears as a ledger and allocation discussion

Not:

Main Menu

↓

Economy

↓

Food

↓

Budget

↓

Numbers

Context comes first.

Menus should appear because something is happening in the world.

---

## Show, Don't Tell

Instead of showing only:

```text
Morale +5
```

Show:

- children playing in the street
- busier markets
- lanterns during festivals
- cleaner roads after investment
- companions reacting to good news
- citizens mentioning recent decisions

Visual feedback creates stronger memories than numbers alone.

---

## Living Interface

The interface should change with the kingdom.

Morning:

- warm colors
- active court
- soft sunlight
- citizens arriving

Rain:

- wet windows
- muted sound
- grey atmosphere
- slower movement

Festival:

- lanterns
- banners
- music
- crowd movement

Crisis:

- urgent companion entrances
- sharper transitions
- reduced decoration
- stronger alert language

UI should feel alive because the kingdom is alive.

---

## Minimal HUD

Players should see the world.

Not windows.

Default gameplay should display only:

- time or day state
- current context
- companion notifications
- important alerts

Everything else appears only when needed.

Persistent UI should be rare.

Important information should be easy to access, but not always visible.

---

# Navigation Philosophy

Players move through people.

Instead of thinking:

```text
Inventory
Technology
Treasury
Events
```

Players should think:

```text
I should ask MP.
I should find Kel.
I should talk with CX.
```

Navigation should be emotional.

Not hierarchical.

The game should avoid a traditional main menu whenever possible.

The Morning Court should function as the player's living home screen.

---

# No Traditional Main Menu

Dream Palace should not begin from a generic menu grid.

The primary hub should be the Morning Court.

The player enters the court.

Companions are present.

Reports are waiting.

The kingdom is already moving.

Systems are accessed through people:

- ask MP about treasury
- ask Kel about technology
- ask Tab about approval status
- ask CX about threats
- ask Mika about culture
- ask Vincey about events
- ask Shian about planning

This makes the interface memorable because every function has a face, voice, and emotional tone.

---

# The Morning Court

Morning Court is both gameplay and interface.

Every day begins here.

Companions report:

- economy
- construction
- investigations
- culture
- public mood
- planning
- unresolved consequences

The court is not a menu background.

It is the center of the player's relationship with the kingdom.

The court should change as the kingdom changes.

Early game:

- fewer companions
- simpler reports
- smaller court presence

Mid game:

- more advisors
- more disagreement
- more simultaneous pressures

Late game:

- larger court
- visible history
- political tension
- deeper companion trust

Players should feel that they built this court over time.

# Camera Philosophy

Camera should communicate emotion.

The camera is part of the interface.

Examples:

Morning Court

↓

wide shot

Discussion

↓

medium shot

Decision

↓

close-up

Companion Reaction

↓

focused reaction shot

World Change

↓

environment shot

Return To Court

↓

wide shot again

Camera movement should make the player feel that they are present inside the scene, not clicking through screens.

---

# Animation Philosophy

Every important interaction deserves motion.

Examples:

- books open
- maps unfold
- blueprints slide across the table
- letters are stamped
- lanterns light up
- mechanisms activate
- evidence is pinned to a board
- festival banners appear in the city
- construction scaffolding changes over time

Motion should support meaning.

Do not animate everything only because it looks active.

---

# Information Hierarchy

Information should be presented in this priority order:

1. Character
2. World context
3. Decision
4. Consequence
5. Numbers

Players should understand why something matters before seeing statistics.

Numbers are important, but they should not be the first emotional impression.

---

# Feedback Philosophy

Every important action should create feedback in at least two layers.

Examples:

## Player funds irrigation

Immediate feedback:

- MP confirms the budget
- Kel begins construction

Visible feedback:

- fields improve
- water channels appear

Later feedback:

- grain increases
- citizens mention better harvests
- MP reports reduced food pressure

## Player ignores a threat

Immediate feedback:

- CX warns again
- Tab questions the risk

Visible feedback:

- border tension increases
- patrols appear

Later feedback:

- a village is attacked
- public trust drops
- Morning Court becomes more urgent

Feedback should connect actions to memory.

---

# Accessibility

Beautiful UI should also be usable.

Support:

- scalable text
- readable contrast
- colorblind modes
- keyboard
- mouse
- controller
- touch in future platforms

Information should never depend only on color.

Important actions should have clear labels and confirmation states.

Motion-heavy sequences should support reduced-motion options.

---

# UI Evolution

The interface should evolve together with the kingdom.

Early game:

- simple court
- fewer companions
- fewer reports
- lighter visual density

Mid game:

- more advisors
- more layered decisions
- more visible history
- stronger faction presence

Late game:

- larger court
- richer environment
- more political tension
- deeper companion reactions

Players should feel that they built the interface naturally by building the kingdom.

---

# UX Test

Ask:

- Can the player understand the situation without reading a wall of text?
- Can they identify the correct companion immediately?
- Does the interface feel like part of the kingdom?
- Does the interaction create a relationship moment?
- Would removing a menu make the experience stronger?

If removing a menu improves immersion, remove the menu.

---

# Anti-Patterns

Avoid:

- floating windows everywhere
- MMORPG hotbar overload
- spreadsheet interfaces
- permanent notification spam
- generic fantasy icons
- menu-first navigation
- detached system panels
- hidden important information
- decorative UI that competes with characters
- text walls that replace performance and staging

Dream Palace should never look like an administrative backend wearing fantasy clothes.

---

# Final Reminder

Players should remember conversations.

Not buttons.

Players should remember advisors.

Not tabs.

---

> **The interface should disappear.**
>
> **The kingdom should remain.**
